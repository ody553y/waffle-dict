### Task A2: Diarization-Aware Transcription Endpoint
**Target:** `worker/waffle_worker/server.py`, `worker/waffle_worker/models.py`

2. Add `speaker: str | None = None` to `TranscriptionSegment`
3. Update `FileTranscriptionResponse.to_dict()` to include speaker fields when present
4. In `WorkerRequestHandler.do_POST` for `/transcriptions/file`:
   - If `request.diarize` is True and diarization pipeline is available:
     a. Transcribe with whisper (get timestamped segments as before)
     b. Run diarization pipeline on the same file
     c. Merge: for each transcription segment, find the overlapping diarization segment and assign its speaker label
     d. Set `speaker` field on each `TranscriptionSegment`
   - If `request.diarize` is True but pipeline unavailable:
     - Return HTTP 400 with `{"error": "diarization_unavailable", "detail": "Speaker diarization requires a HuggingFace token. Set HF_TOKEN or pass --hf-token."}`
   - If `request.diarize` is False: existing behavior unchanged
5. Merge algorithm (in a new `worker/waffle_worker/merge.py`):
   ```python
   def merge_speakers(
       transcription_segments: list[TranscriptionSegment],
       diarization_segments: list[DiarizationSegment],
   ) -> list[TranscriptionSegment]:
   ```
   - For each transcription segment, compute the midpoint `(start + end) / 2`
   - Find the diarization segment whose time range contains the midpoint
   - If no overlap, use the nearest diarization segment
   - Assign that diarization segment's `speaker` label
   - Return new list with speaker labels populated
6. Add `GET /diarization/status` endpoint:
   - Returns `{"available": true/false, "model": "pyannote/speaker-diarization-3.1"}` so the Swift app can check availability
7. Pass `DiarizationPipeline` to `WorkerHTTPServer`:
   - Add `diarization_pipeline: DiarizationPipeline | None` field
   - Thread it through `make_server()` and `serve()`

**Notes:**
- Diarization adds latency. For a 1-minute audio file, expect ~5–15 seconds extra on Apple Silicon.
- The merge is approximate — pyannote segments and whisper segments won't align perfectly. Midpoint matching is good enough.
- Keep diarization optional. The default is `diarize: false` so existing behavior is unchanged.

**Tests:** Unit test merge algorithm with known segments (exact overlap, partial overlap, no overlap). Test endpoint returns 400 when diarization unavailable. Test that `diarize: false` requests work unchanged. Test `/diarization/status` endpoint.

## Track B: Signed Model Distribution

### Task B1: Ed25519 Manifest Signing Tool
**Target:** New `tools/sign-manifest.py`

Create a CLI tool that signs the model manifest with Ed25519 for tamper-proof model distribution.

1. Create `tools/sign-manifest.py`:
   ```
   Usage:
     sign-manifest.py generate-keys --output <dir>
     sign-manifest.py sign --manifest <path> --private-key <path> --output <path>
     sign-manifest.py verify --manifest <path> --public-key <path>
   ```
2. Key generation (`generate-keys`):
   - Generate Ed25519 keypair using `cryptography` library
   - Write `manifest-signing-key.pem` (private) and `manifest-signing-key.pub` (public, raw 32 bytes hex-encoded)
   - Print public key hex to stdout for embedding in the Swift app
3. Signing (`sign`):
   - Read manifest JSON
   - Normalize: compact JSON with sorted keys (deterministic serialization)
   - Sign the normalized bytes with Ed25519 private key
   - Output a signed envelope:
     ```json
     {
       "manifest_version": 2,
       "signed_at": "2026-03-30T12:00:00Z",
       "signature": "<hex-encoded Ed25519 signature>",
       "models": [ ... original manifest array ... ]
     }
     ```
4. Verification (`verify`):
   - Read signed manifest
   - Extract `models` array, re-normalize to compact sorted JSON
   - Verify `signature` against the normalized bytes using public key
   - Exit 0 on valid, exit 1 on invalid, with human-readable output
5. Add `cryptography>=43.0` to `tools/requirements.txt` (separate from worker requirements)

**Notes:**
- Ed25519 chosen over RSA for compact signatures (64 bytes) and fast verification.
- The private key stays with the release maintainer. Only the public key ships in the app.
- Deterministic serialization is critical — `json.dumps(models, sort_keys=True, separators=(',', ':'))`.
- The `manifest_version: 2` field lets the Swift app distinguish signed vs unsigned manifests.

**Tests:** Unit test key generation, signing, and verification round-trip. Test that tampering with any field invalidates the signature. Test that the tool exits correctly for valid/invalid manifests.

**Acceptance:** `sign-manifest.py generate-keys` produces a valid keypair. `sign-manifest.py sign` produces a signed manifest. `sign-manifest.py verify` confirms valid signatures and rejects tampered ones.

### Task B2: Signed Manifest Verification in Swift
**Target:** `Sources/WaffleCore/ModelCatalog.swift`, `Sources/WaffleCore/Resources/models-manifest.json`

Add Ed25519 signature verification to the Swift model catalog.

1. Update `models-manifest.json` to the signed envelope format:
   ```json
   {
     "manifest_version": 2,
     "signed_at": "2026-03-30T12:00:00Z",
     "signature": "<hex>",
     "models": [ ... existing array ... ]
   }
   ```
2. Define `SignedModelManifest` in `ModelCatalog.swift`:
   ```swift
   struct SignedModelManifest: Codable {
       let manifestVersion: Int
       let signedAt: String
       let signature: String
       let models: [ModelCatalogEntry]
   }
   ```
3. Add `ManifestVerifier` using CryptoKit:
   ```swift
   struct ManifestVerifier {
       static let publicKeyHex = "..." // 32-byte Ed25519 public key, hex-encoded
       func verify(manifest: SignedModelManifest) throws -> Bool
   }
   ```
   - Decode public key from hex → `Curve25519.Signing.PublicKey`
   - Reconstruct normalized JSON from `manifest.models` (compact, sorted keys — must match Python's normalization)
   - Verify `signature` (hex → Data) against normalized bytes
   - Throw `ManifestVerificationError.invalidSignature` on failure
4. Define `ManifestVerificationError`:
   - `invalidSignature` — signature doesn't match
   - `unsupportedVersion` — `manifest_version` is not recognized
   - `malformedManifest` — can't parse envelope
5. Update `ModelCatalogService.loadCatalog()`:
   - Attempt to decode as `SignedModelManifest` first
   - If `manifest_version == 2`, verify signature before returning `models`
   - If verification fails, throw `ManifestVerificationError.invalidSignature`
   - Backwards compatibility: if decoding as signed manifest fails, try legacy `[ModelCatalogEntry]` array format (for development builds)
6. Add `CodingKeys` to `SignedModelManifest`:
   - `manifestVersion` → `"manifest_version"`
   - `signedAt` → `"signed_at"`

**Notes:**
- CryptoKit's `Curve25519.Signing` is available on macOS 14+ — already our minimum target.
- JSON normalization must be byte-identical between Python and Swift. Use `JSONEncoder` with `sortedKeys` and no pretty printing. Test cross-language normalization explicitly.
- The public key is compiled into the binary. Rotating the key requires an app update.
- Development builds can bypass verification by decoding as the legacy unsigned array.

**Tests:** Test signature verification with a known keypair + manifest. Test that a tampered manifest is rejected. Test backwards-compatible loading of unsigned manifest. Test cross-format normalization produces identical bytes.

**Acceptance:** Signed manifest loads and verifies successfully. Tampered manifests are rejected. Legacy unsigned manifests still load in dev.

### Task C2: Notarization Workflow
**Target:** `scripts/build-release.sh` (extend), new `scripts/notarize.sh`

Add Apple notarization so the app can be distributed outside the Mac App Store.

1. Create `scripts/notarize.sh`:
   ```bash
   #!/bin/bash
   set -euo pipefail
   APP_PATH="build/Waffle.app"
   ZIP_PATH="build/Waffle.zip"

   # Create ZIP for notarization
   ditto -c -k --keepParent "$APP_PATH" "$ZIP_PATH"

   # Submit for notarization
   xcrun notarytool submit "$ZIP_PATH" \
       --apple-id "$APPLE_ID" \
       --team-id "$TEAM_ID" \
       --password "$APP_SPECIFIC_PASSWORD" \
       --wait

   # Staple the notarization ticket
   xcrun stapler staple "$APP_PATH"
   ```

   ```bash
   # After notarization + stapling:
   hdiutil create -volname "Waffle" -srcfolder "$APP_PATH" \

   # Sign the DMG too
   codesign --force --sign "$SIGNING_IDENTITY" "build/Waffle-${VERSION}.dmg"
   ```
3. Environment variables required:
   - `SIGNING_IDENTITY` — Developer ID Application certificate name

## Notes for Codex

- **Swift version:** 6.0+ strict concurrency throughout. New types must be `Sendable` where appropriate.
- **Test framework:** Swift Testing (`import Testing`, `@Test`, `#expect`) — NOT XCTest.
- **Python test framework:** `unittest` — match existing worker test patterns.
- **New Swift dependency:** Sparkle 2.6+ (Track C3 only). No other new Swift packages.
- **New Python dependency:** `pyannote.audio>=3.1` for diarization (Track A only). Requires `torch`.
- **Worker stays at `127.0.0.1:8765`** — same REST protocol, extended with `diarize` field and `/diarization/status` endpoint.
- **Ed25519 in Swift:** Use CryptoKit `Curve25519.Signing` — available macOS 14+.
- **Ed25519 in Python:** Use `cryptography` library `Ed25519PrivateKey`/`Ed25519PublicKey`.
- **Sparkle Ed25519 key is separate** from the manifest signing Ed25519 key. Don't reuse keys across different signing purposes.
- **JSON normalization for manifest signing:** Python: `json.dumps(models, sort_keys=True, separators=(',', ':'))`. Swift: `JSONEncoder` with `outputFormatting = [.sortedKeys]` and no pretty printing. Cross-language byte-equality is mandatory.
- **Database migrations:** Append new migrations after existing ones. No schema changes needed for Phase 5 (speaker data lives inside JSON segments column).
